# LinkHub

LinkHub 是一个面向技术内容创作者、数字资源管理者和开发者的轻量级外链聚合与导航系统。该项目并非一个传统的爬虫或采集平台，而是一个专注于人工精选与高质量分类的技术资源目录，帮助用户在信息过载的环境中快速定位有价值的工具、文档与社区。LinkHub 定位于个人与小型团队的内部知识库外延，通过结构化的数据组织方式，将零散的优质外部链接转化为可复用、可分享、可维护的资产。

LinkHub 的核心价值在于其简洁的架构与明确的边界。项目本身不存储任何第三方内容，仅保留链接元数据与分类标签，所有实际资源均指向原始出处。这种方式既规避了版权与存储成本问题，又使得项目天然具备轻量化与高可移植性。无论是用于个人书签的公开备份，还是作为团队内部的技术周报汇总，LinkHub 都能够提供一个清晰、快速、无干扰的访问入口。

## 功能概览

- **多级分类目录**：支持无限层级的分类嵌套，允许用户按照技术栈、使用场景或工作流程自定义组织逻辑，每个分类可独立配置展示顺序与图标标识。

- **链接元数据管理**：每条链接支持记录标题、描述、标签、添加日期、访问次数与状态标记（有效/失效/维护中），并预留了自定义字段以供扩展。

- **全文检索与过滤**：基于标题、描述和标签的快速关键词检索，同时支持按分类、按状态、按时间范围进行多维度过滤，方便在大量链接中精准定位。

- **批量导入与导出**：支持通过 CSV 和 JSON 格式批量导入链接数据，并支持完整导出为结构化文件，便于迁移、备份或与其他系统集成。

- **公开与私有混合模式**：允许将部分分类或链接标记为私有，仅对登录用户可见，其余内容可作为公开导航页发布，适用于内部知识共享与对外展示并存的场景。

- **访问统计与失效检测**：自动记录每个外链的点击次数，并支持定时任务对链接进行可用性探测，主动标记失效链接，减少无效跳转。

- **响应式主题与自定义样式**：内置三套不同风格的响应式主题，同时支持通过 CSS 变量进行简单的颜色与布局定制，适配个人博客、团队文档站等多种嵌入环境。

## 应用场景

- **技术团队内部知识导航**：研发团队可将日常开发中积累的 API 文档、开源库主页、运维手册、设计规范等外部资源统一收录至 LinkHub，替代浏览器收藏夹的碎片化管理，新成员入职时可快速了解团队常用的工具链。

- **个人技术博客的友情链接与推荐阅读**：技术博主可以使用 LinkHub 作为博客侧边栏或独立页面的外链聚合模块，向读者展示经过筛选的优质学习资源、行业资讯源以及合作伙伴站点，提升博客的信息附加价值。

- **开源项目文档的外部参考附录**：开源项目的维护者可以在项目文档中嵌入 LinkHub 页面，集中列出项目依赖的第三方工具、参考规范、社区论坛以及相关生态项目，帮助使用者快速建立上下文认知。

- **线上课程或技术培训的延伸阅读清单**：讲师或培训组织者可利用 LinkHub 按课程章节整理配套的阅读材料、视频链接、在线练习环境等，学员通过统一入口即可获取全部课外资源，避免因链接分散而影响学习体验。

## 快速开始

以下步骤将在本地环境中完成 LinkHub 的克隆、依赖安装与开发服务器启动。

```bash
# 克隆代码仓库
git clone https://github.com/your-org/linkhub.git
cd linkhub

# 安装项目依赖
npm install

# 启动本地开发服务，默认监听 3000 端口
npm run dev
```

启动成功后，在浏览器中访问 http://localhost:3000 即可进入 LinkHub 的管理控制台。首次启动时，系统会自动生成一个默认的管理员账户，用户名与密码将在控制台日志中输出，请及时登录并修改默认凭证。如需构建生产环境静态文件，请执行 `npm run build` 命令，构建产物将输出至 `dist` 目录。

## 安装要求

| 依赖 | 必需 | 说明 |
|---|---|---|
| Node.js >= 18.0.0 | 是 | 项目运行时环境，推荐使用 LTS 版本，建议同时安装 npm 或 yarn 作为包管理器 |
| SQLite 3.x | 是 | 默认内嵌数据库，无需额外安装，用于存储链接、分类及用户数据，适合小型部署场景 |
| PostgreSQL >= 12.0 | 否 | 生产环境推荐使用，可通过环境变量切换数据源，需自行安装并创建数据库实例 |
| Redis >= 6.0 | 否 | 可选缓存层，用于提高频繁读取场景下的响应速度，非必需，默认关闭 |
| Nginx 或 Apache | 否 | 生产环境反向代理配置建议，用于处理静态资源缓存与负载均衡，非必需但推荐 |
| Git >= 2.20 | 否 | 仅当通过源码方式部署时需要，用于拉取代码仓库，Docker 部署方式可跳过 |
| Docker Engine >= 20.10 | 否 | 若采用容器化部署方式，则需要 Docker 环境，并支持 Docker Compose 编排 |
| 可用域名与 SSL 证书 | 否 | 生产环境如提供公网访问，建议配置 HTTPS，但不强制，项目本身兼容 HTTP 协议 |

## 贡献指南

1. 在 GitHub 上 Fork 本仓库至个人账号，并克隆到本地开发环境。请确保本地 Node.js 版本满足要求，并正确安装所有依赖。建议在开发前阅读 `docs/developer-guide/architecture.md` 以了解整体设计思路。

2. 新建一个功能分支，分支名称请遵循 `feature/功能简述` 或 `fix/问题简述` 的格式。所有代码提交应附带清晰的 commit message，说明变更原因与影响范围。提交前请运行 `npm run lint` 和 `npm run test` 确保代码风格与测试用例均通过。

3. 若新增或修改了 API 接口，请同步更新 `docs/developer-guide/api-reference.md` 中的对应说明。若调整了前端组件或主题变量，需同步更新 `docs/developer-guide/frontend-theming.md` 文档。文档与代码保持同步是合并请求的基本要求。

4. 完成开发后，将分支推送至个人远程仓库，并通过 GitHub 界面发起 Pull Request 至本仓库的 `main` 分支。请在 PR 描述中详细列出变更内容、测试覆盖情况以及是否涉及数据库迁移或环境变量变更。

5. 项目维护者将在一周内对 PR 进行评审，如有修改意见会在 PR 下留言。合并后，您的贡献将被收录至项目贡献者列表，并可在 CHANGELOG 中看到对应的更新记录。

## 常见问题

**问：LinkHub 是否可以完全离线部署，不依赖任何外部 CDN 或远程服务？**

答：可以。前端构建时会将所有依赖打包至本地静态文件中，不强制加载任何外部 CDN 资源。但请注意，如果您的链接数据中包含指向外部站点的 URL，访问这些链接时仍需网络连接。项目本身的管理界面和核心功能在完全内网环境下均可正常运行。

**问：如何从旧版本迁移数据，或者在不同数据库之间迁移？**

答：项目内置了数据导出与导入命令行工具。您可以通过 `npm run export -- --format=json` 导出当前数据库全部内容为 JSON 文件，然后在新环境或新版本中执行 `npm run import -- --file=data.json` 完成恢复。跨数据库迁移（如 SQLite 转 PostgreSQL）时，同样使用该方式，系统会自动适配目标数据库的 schema。

**问：LinkHub 的链接失效检测机制是否会影响页面性能？**

答：失效检测通过独立的后台定时任务执行，默认每 24 小时运行一次，且仅对过去 7 天内未被检测的链接发起 HEAD 请求。该任务不会阻塞主应用的请求处理流程。如需实时检测，可在管理后台手动触发单条链接的检测，该操作为异步执行，不会导致页面长时间等待。
