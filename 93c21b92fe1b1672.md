# LinkVault

LinkVault 是一个面向技术团队与内容运营者的轻量级外链资源聚合与管理平台。其核心定位并非传统的书签管理工具，而是一个基于标签体系与健康度检测的“活”链接集散中心。项目旨在解决技术文档、开源项目 README 以及团队内部知识库中外部链接日益膨胀、缺乏分类、且频繁失效难以维护的痛点。LinkVault 通过预设的抓取规则与状态监控，帮助用户将散落的优质技术资源、工具站点与官方文档进行结构化归档，并以可嵌入的挂件形式对外输出为整洁的导航页，从而替代杂乱的收藏夹或低可读性的纯文本 URL 列表。

本项目面向需要维护高密度外部引用资源的开源项目维护者、技术社区版主、以及企业内部的平台工程团队。通过对链接进行语义化分组、访问速度探测与可用性标记，LinkVault 提供了一套不依赖数据库、基于静态配置即可运行的轻量化方案，适用于部署在低成本的边缘计算环境或静态托管服务上。

## 功能概览

- **批量链接导入与自动清洗**：支持从纯文本、CSV 或 Markdown 列表中批量摄入 URL，自动去除重复项、识别协议缺失并补全，同时对包含中文或特殊字符的链接进行 Encode 处理，确保入库数据的格式标准性。

- **基于标签树的动态分类**：允许用户为每条链接配置多级标签（例如 `工具/网络诊断`、`文档/前端/React`），系统自动生成分类树视图。分类变更时，所有挂载页面将实时同步，无需手动迁移链接。

- **链接健康度被动探测**：集成简易的 HTTP HEAD 请求模块，可依据可配置的时间间隔（如每日凌晨）对库内链接进行可达性检测。响应码异常或超时链接将被标记为“存疑”状态，并记录最近一次检测时间与状态码，便于管理员快速清理或替换失效资源。

- **自定义外链门户生成器**：基于 Handlebars 或 EJS 模板引擎，用户可选择需要公开的分类与标签，一键生成纯静态的 HTML 导航页面。生成的页面支持响应式布局与暗色主题，可直接挂载到 GitHub Pages、Nginx 或对象存储服务中对外分发。

- **配置即代码的全量导出**：所有链接数据、标签结构与检测配置均可导出为单一的 YAML 或 JSON 文件，支持版本化管理。团队成员可通过 Pull Request 方式对链接库进行增删改，变更历史清晰可溯。

- **嵌入用的轻量级小部件脚本**：提供一段可嵌入其他网站的 JavaScript 片段，允许第三方页面以 iframe 或动态渲染方式引用 LinkVault 的指定分类列表，解决外部资源跨站引用与样式隔离的问题。

## 应用场景

- **开源项目外链资料页**：许多开源项目在 README 中附带大量“生态系统”、“相关工具”或“学习资源”链接。随着项目迭代，这些链接容易过时。维护者可在仓库的 `docs` 目录下维护一份 LinkVault 配置文件，通过 CI 自动生成并发布外链页面，确保文档中的引用链接始终可查、可验证。

- **技术社区知识库导航**：技术问答社区或内部 Wiki 常设有“资源汇总”板块。LinkVault 允许社区版主将零散的论坛置顶帖、精华博文链接、官方 SDK 下载地址等统一收录，并按“入门”、“进阶”、“运维”、“安全”等维度分类，最终输出为社区导航栏中的固定菜单项，降低新成员的信息检索成本。

- **企业内部工具链门户**：研发团队通常同时使用 Jira、Confluence、GitLab、Jenkins、ArgoCD 等多套内部系统。LinkVault 可作为这些内部工具的入口聚合页，由运维人员集中维护域名与访问端口变更。当某套系统迁移 IP 或更换域名时，只需在 LinkVault 配置中修改一处，所有引用该入口的文档或仪表盘即同步更新。

- **个人技术写作的引用仓库**：技术博主或独立撰稿人在撰写系列文章时，经常需要反复引用相同的规范文档、API 参考或统计数据来源。使用 LinkVault 为每篇文章建立独立的标签集合，可保证同一来源在不同文章中的表述与 URL 完全一致，同时当源站改版时只需更新配置中的链接目标，无需逐一修改每篇文章。

## 快速开始

以下指令适用于 Linux / macOS 环境，假设系统已安装 Git、Node.js 18+ 与 pnpm 包管理器。

```bash
# 克隆项目主仓库
git clone https://github.com/example/linkvault.git
cd linkvault

# 安装项目依赖（包含核心解析引擎与检测工具）
pnpm install

# 启动开发服务器，默认监听 3000 端口，访问 http://localhost:3000 进入配置面板
pnpm run dev
```

对于生产环境部署，建议执行 `pnpm run build` 构建静态资源，然后使用 `pnpm run start` 启动基于 Fastify 的正式服务进程。若需以 Docker 方式运行，项目根目录包含 `Dockerfile` 与 `docker-compose.yml` 示例，可直接使用 `docker compose up -d` 一键启动。

## 贡献指南

我们欢迎并鼓励社区以多种形式参与 LinkVault 项目的改进。请遵循以下流程以保证协作效率：

1.  **提交议题 (Issue)**：在发现缺陷或提出新功能需求时，请先检索现有议题列表以避免重复。新建议题时请使用提供的模板，清晰描述复现步骤、预期行为与实际行为，并附上相关日志或配置文件片段。

2.  **分支开发规范**：请从 `main` 分支派生新的特性分支，分支命名建议采用 `feat/`、`fix/` 或 `docs/` 前缀加简要描述（例如 `feat/redis-cache-support`）。所有开发工作应在该分支上完成，并确保代码通过本地单元测试。

3.  **代码风格与检查**：提交前请运行 `pnpm run lint` 和 `pnpm run format` 以自动格式化代码。项目使用 ESLint 与 Prettier 统一风格，CI 流程将拒绝不符合规范格式的合并请求。

4.  **编写或更新测试**：对于新增功能或修复，请提供对应的测试用例。单元测试应覆盖核心逻辑分支，集成测试应验证 API 契约与模板渲染结果。测试覆盖率不应低于当前主干水平。

5.  **发起合并请求 (Pull Request)**：推送分支至远程仓库后，向 `main` 分支发起 PR。PR 描述中需关联对应的议题编号，并简要说明变更内容与验证方式。至少需要一名维护者进行 Code Review 后方可合并。

## 常见问题

**问：LinkVault 是否支持从浏览器书签文件（HTML 格式）直接导入？**

答：目前官方版本尚未内置对 Netscape 书签格式的解析器。但您可以将书签文件中的链接与注释提取为 CSV 格式，然后使用项目 `scripts/import-csv.sh` 辅助脚本进行批量导入。我们计划在后续版本中增加对主流浏览器书签导出格式的原生支持。

**问：健康度探测会对外部站点造成压力吗？如何调整频率？**

答：健康探测仅发送单个 HTTP HEAD 请求并等待响应头，不会下载完整页面内容，单次请求开销极小。默认探测间隔为 24 小时，且支持配置随机抖动（Jitter）以避免所有链接在同一时刻触发请求。您可以在 `config/default.yaml` 中调整 `checker.interval` 参数（单位为分钟），最低建议间隔不低于 60 分钟。

**问：生成的静态导航页能否完全离线访问？**

答：默认生成的 HTML 页面包含内联 CSS 样式与字体图标 Base64 编码，不依赖外部 CDN 资源，因此可在完全隔离的内网环境中正常浏览。但若您启用了“外部推荐链接”模块或使用了非本地的统计脚本，则需根据实际模板配置判断。
