# CloudNavigator

CloudNavigator 是一个面向技术从业者与开源爱好者的高质量网络资源导航与聚合平台。本项目致力于系统化收录互联网中与云计算、边缘计算、开源社区、技术文档及开发者工具相关的优质外部链接，并通过清晰的结构与分类帮助用户快速定位所需信息。不同于传统的书签管理工具，CloudNavigator 本身不存储任何实际内容，而是以目录索引与注释说明的方式，将散落在各处的优质技术站点整理为可检索、可分享的知识图谱。

本项目主要服务于运维工程师、全栈开发人员、技术决策者以及开源项目贡献者。在日常工作中，用户往往需要频繁查阅不同厂商的技术文档、访问多个社区论坛或跟踪开源项目的最新动态，而零散的书签与模糊的搜索引擎结果会显著降低效率。CloudNavigator 通过预设的分类体系与人工筛选标准，确保每个收录链接均具有明确的领域归属与实际使用价值，从而将信息检索成本降至最低。同时，项目本身采用纯静态 Markdown 渲染方式，无需数据库与后端服务，可轻松部署于任何支持静态页面的托管平台，亦可供其他开源项目作为子模块引用。

## 功能概览

- **精准分类索引**：所有收录资源按技术领域、服务类型、内容形态进行三级标签划分，支持多维度筛选与快速定位。

- **人工质量审核**：每个链接在收录前均经过可用性与内容相关性验证，定期清理失效链接与过期内容，确保长期有效性。

- **轻量级本地部署**：项目仅依赖标准的 Markdown 解析器与静态 Web 服务器，无需构建工具链或运行时环境，开箱即用。

- **可扩展的数据结构**：采用 YAML 格式维护外部链接元数据，支持用户通过 Pull Request 方式提交新增资源，便于社区共同维护。

- **内置变更日志**：每次更新均记录时间戳与变更摘要，用户可通过提交历史追踪资源列表的演进过程，便于审计与回溯。

- **多视图展示模式**：既提供面向快速检索的列表视图，也提供面向深度阅读的卡片视图，适应不同用户的使用习惯。

- **项目内交叉引用**：在各章节及文档内部提供相关资源的交叉引用链接，帮助用户在同一上下文中发现更多关联内容。

- **环境无关性设计**：所有资源 URL 均以原始字符串形式存储，不做协议补全或域名规范化处理，保留用户原始输入信息。

## 应用场景

- 新项目技术选型阶段，架构师需要快速对比多家云服务商的产品特性与文档质量。CloudNavigator 预先整理主流云平台的官方入口与开发者指南，节省逐一手动搜索的时间。

- 开源社区维护者在编写项目的 README 或贡献指南时，需要引用外部资源作为参考材料。可直接使用 CloudNavigator 中的稳定链接，避免因个人书签变更导致的文档失效问题。

- 技术培训或企业内部知识库搭建过程中，管理人员希望建立一个受控且可控的外部资源白名单。CloudNavigator 的分类结构与审核机制可作为基础模板，方便定制化裁剪与扩展。

## 快速开始

以下操作假设您已安装 Git 与标准的 POSIX 兼容 Shell 环境（如 Linux、macOS 或 Windows WSL）。执行命令前请确保当前目录具有写入权限。

```bash
# 克隆项目仓库至本地
git clone https://github.com/cloudnavigator/cloudnavigator.git

# 进入项目根目录，安装依赖（仅需复制静态资源与解析工具）
cd cloudnavigator
cp -r src/assets/ ./public/ && cp src/parser.js ./public/

# 启动内置的简易静态服务器，默认监听端口 8080
python3 -m http.server 8080 --directory ./public
```

启动后，在浏览器中访问 `http://localhost:8080` 即可浏览已收录的资源列表。若需更新资源数据，直接编辑 `src/data/resources.yaml` 文件后刷新页面即可生效。

## 贡献指南

1. 在 GitHub 上 Fork 本仓库至您的个人账户，并在本地克隆 Fork 后的副本。请确保您的开发分支基于最新的 main 分支创建，分支命名建议使用 `feature/xxx` 或 `fix/xxx` 格式。

2. 编辑 `src/data/resources.yaml` 文件，添加新资源条目或修改现有条目。每个条目必须包含 `url`、`title`、`category` 和 `description` 字段，格式请参照已有的示例条目。新增的 URL 必须为原始字符串，不得包含 HTML 标签或 Markdown 链接语法。

3. 运行本地健康检查脚本确认所有现有链接仍可访问，并执行 `tests/` 下的单元测试确保数据格式未被破坏。若脚本因网络问题失败，请先排除环境因素后再提交。

4. 提交变更时，请编写清晰且语义化的提交信息，格式为 `<类型>: <简短描述>`，例如 `feat: 添加三家云服务商的官方文档链接`。提交后推送到您的 Fork 仓库。

5. 通过 GitHub 界面发起 Pull Request 至本仓库的 main 分支，并在 PR 描述中说明变更目的与测试结果。维护者将在 3 个工作日内审核，必要时会提出修改意见，合并后您的贡献将出现在下一版发布中。

## 常见问题

**Q: 为什么某些链接无法在项目内直接打开或显示内容？**

A: CloudNavigator 仅作为外部链接的索引与分类展示平台，并不代理或缓存任何第三方内容。如果您遇到链接无法访问的情况，可能是目标站点临时下线、网络环境限制或域名解析问题。建议先复制 URL 并在浏览器中单独访问，若仍无法打开，可通过 GitHub Issues 向我们报告，维护者会定期核查并移除失效资源。

**Q: 我想提交新的资源链接，但不知道如何选择合适的分类和标签？**

A: 我们建议您先浏览 `resources.yaml` 中已有的分类体系，将新链接归入最匹配的现有类别。如果确实找不到合适的分类，可以在 Pull Request 中提出新增类别的建议，并在描述中说明理由。维护者会评估分类的必要性与通用性，若合理则会合并并同步更新文档中的分类说明。请注意，我们不接受商业广告或纯推广性质的非技术内容。
