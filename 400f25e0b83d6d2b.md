# LinkPilot

LinkPilot 是一个面向技术内容创作者、开源项目维护者以及互联网资源管理者的轻量级外链资源汇总与导航平台。该项目并非传统的网址收藏夹或书签管理器，而是一个强调可维护性、可审计性与团队协作的静态外链数据中台。其核心目标在于帮助用户从碎片化的浏览器收藏夹与零散的文档碎片中解脱出来，以结构化的数据视图统一管理项目依赖的外部文档、社区阵地、镜像源以及各类技术服务入口。

LinkPilot 主要解决开源项目文档中“外链失效”、“入口分散”以及“新人上手路径不清晰”三大痛点。通过将外链资源与项目版本进行关联绑定，项目维护者可以清晰追溯每个外部资源在特定版本中的必要性与可用状态。同时，项目内置的链接健康检查报告生成能力，使得定期清理死链、更新迁移地址成为一项可自动化执行的工程任务，而非依赖人工记忆的繁琐杂务。LinkPilot 适合作为中大型开源项目的官方文档外链附录的生成器，也适合作为技术社区的内容聚合入口。

## 功能概览

- **结构化外链数据建模** 提供基于 YAML 与 JSON Schema 的双模数据定义方式，支持为每条外链添加分类标签、可用性状态、维护人、最后检查时间等元数据字段，确保外链信息具备工程级可维护性。

- **多格式静态站点生成** 内置 Markdown、HTML 与 JSON 三种输出引擎，用户可通过单条命令将外链数据集渲染为适用于 GitHub README 的 Markdown 表格、适用于项目官网的 HTML 卡片布局，或适用于 API 调用的 JSON 数据结构。

- **链接可用性自动巡检** 集成异步 HTTP 探测引擎，支持批量检测外链的响应状态码、页面标题一致性以及 SSL 证书有效期，检测结果以差异报告形式输出，便于定位失效或异常资源。

- **版本快照与变更审计** 每条外链的新增、修改与删除操作均记录操作者与时间戳，支持按版本标签（如 v2.1.0）导出该版本对应的完整外链快照，确保历史文档引用的外部资源可追溯。

- **外链关系图谱可视化** 基于外链之间的引用关系与分类共现频次，生成力导向关系图，帮助项目维护者识别核心资源节点与潜在的单点依赖风险。

- **团队协作与审核工作流** 提供基于 Pull Request 模式的外链变更提案机制，变更内容需经过指定的审核人确认后方可合并至主分支，适用于企业级内部文档库的外链治理场景。

## 应用场景

- 开源项目官网的社区生态页生成。项目维护者可以将社区论坛、第三方教程、插件市场、镜像下载站等入口统一录入 LinkPilot，每次发布新版本时自动生成对应的生态资源附录，随官网一同部署，确保用户始终能够通过官方入口访问到经过验证的外部资源。

- 技术文档团队的文档引用源管理。当文档中需要引用外部规范、SDK 下载地址或 API 参考链接时，团队可将这些链接统一托管于 LinkPilot 并标注预期可用期限。每月自动生成的巡检报告会列出即将失效或已失效的链接，文档工程师据此及时更新文档内容，避免用户因访问到过期链接而产生困惑。

- 企业内部开发者门户的导航聚合。企业内的平台工程团队可以使用 LinkPilot 聚合内部 GitLab 入口、CI/CD 系统地址、监控面板、日志查询工具、内部 NPM 镜像等数十个内部服务入口，通过版本快照功能记录每次服务迁移期间的入口变更历史，便于事后回溯与排障。

## 快速开始

以下命令将克隆项目仓库、安装依赖并启动本地开发服务，默认监听 3000 端口。

```bash
git clone https://github.com/linkpilot-org/linkpilot-core.git
cd linkpilot-core
npm install
npm run dev
```

执行完上述命令后，访问控制台输出的本地地址即可进入 LinkPilot 的数据管理界面。首次启动时，系统会自动生成示例外链数据集与默认配置文件，用户可基于示例数据快速熟悉数据结构与操作流程。

## 贡献指南

1. 查阅 Issue 列表与项目看板，确认当前待处理的任务或提议的新功能未被他人认领。对于较大规模的改动，建议先在 Issue 中描述设计方案与预期影响，获得核心维护者反馈后再投入开发。

2. 派生项目仓库至个人账户，基于 main 分支创建功能分支，分支命名建议采用 feat/功能简述 或 fix/问题简述 格式，确保分支名称能够清晰表达变更意图。

3. 完成代码开发后，请补充或更新对应的单元测试用例与文档说明。新增外链数据字段或输出格式时，需同步更新 samples/ 目录下的示例数据以及 docs/ 目录下的相关文档。

4. 提交代码前运行完整的测试套件，确保所有已有测试用例保持通过状态。提交信息请遵循约定式提交规范，即使用 feat:、fix:、docs:、chore: 等前缀描述变更类型。

5. 发起 Pull Request 至主仓库的 main 分支，PR 描述中需说明变更的背景、实现方式以及测试覆盖情况。PR 需经过至少一名项目维护者的代码审查，审查通过后方可合并。

## 常见问题

**Q：巡检报告显示某个链接返回 403 状态码，但浏览器访问正常，应如何排查？**

A：该情况通常由目标站点对 User-Agent 或 Referer 请求头做限制导致。LinkPilot 的探测引擎默认使用 Node.js 原生 http 模块的请求头，部分安全策略严格的站点会拒绝非浏览器请求。解决方案为在配置文件中自定义探测请求头，将 User-Agent 模拟为常见浏览器字符串。同时建议检查目标站点是否存在访问频率限制，可适当调大探测间隔时间。

**Q：项目支持从现有浏览器收藏夹或书签文件导入外链数据吗？**

A：LinkPilot 本身不直接内置浏览器书签解析器，但提供了通用的 JSON 导入接口。用户可将浏览器书签导出为 HTML 文件后，使用社区提供的书签解析工具转换为符合 LinkPilot Schema 的 JSON 数据格式，再通过数据导入命令完成批量录入。后续版本规划中将优先支持 Chrome 和 Firefox 书签文件的直接导入能力。
