# HrefHub

HrefHub 是一个面向技术内容创作者、开源文档维护者以及网络资源整理者的轻量级外链资源聚合与导航工具。该项目不提供任何实质内容存储或代理服务，仅作为结构化外部链接的索引与展示层，帮助用户快速定位分散在多个域名下的可用资源节点。HrefHub 适用于需要频繁更新外部参考链接的技术文档站、个人知识库或社区维基，通过集中化管理外部 URL 降低链接失效与维护成本。

## 功能概览

- **多源外链统一索引**：支持将来自不同域名、不同协议的外部链接按自定义分类聚合，生成可维护的链接清单，便于批量校验与更新。

- **纯静态站点生成**：基于配置文件生成完全静态的 HTML 页面，无后端依赖，可部署于任意 Web 服务器或 CDN，确保高可用性与低延迟。

- **链接状态自动检测**：内置周期性链接可达性检查模块，能够标记失效链接并生成报告，辅助管理员及时替换或移除不可用资源。

- **标签与全文检索**：为每条外链附加标签与简短描述，支持基于关键词的即时搜索，提升在大量链接中定位目标资源的效率。

- **响应式展示与暗色主题**：前端界面适配桌面与移动设备，并内置暗色与亮色两套主题，满足不同阅读环境下的视觉需求。

- **导入导出与备份**：支持以 JSON 或 YAML 格式批量导入导出链接数据，便于迁移、版本控制或与其他工具集成。

## 应用场景

- **技术文档维护**：技术文档中常引用外部规范、SDK 下载页或社区讨论帖，HrefHub 可将这些分散引用统一托管，文档内仅保留指向 HrefHub 条目的短引用，当外部链接变化时只需在 HrefHub 中更新一处。

- **个人知识库外链管理**：知识库笔记中大量保存参考链接，使用 HrefHub 作为独立的外链抽屉，笔记内只记录锚点，既能保持笔记内容精简，又能批量检查链接有效性。

- **社区资源导航页**：开源社区或线上兴趣小组可部署 HrefHub 作为社区首页，集中展示成员推荐的教程、工具站、镜像源等外部资源，降低新成员的信息获取门槛。

- **项目 README 外链补充**：GitHub 项目 README 不宜堆砌过多外部链接，可将 HrefHub 生成的链接列表页作为 README 中「相关资源」章节的延伸，保持 README 简洁且易于维护。

## 快速开始

以下命令适用于 Linux / macOS / Windows WSL 环境，假设系统已安装 Git 与 Node.js 18+。

```bash
# 克隆仓库
git clone https://github.com/hrefhub/hrefhub.git
cd hrefhub

# 安装依赖
npm install

# 生成静态站点（基于 config/links.yaml）
npm run build

# 启动本地预览服务
npm run serve
```

执行完成后，访问控制台输出的本地地址（通常为 http://localhost:8080 ）即可查看生成的导航页面。若需自定义链接数据，请编辑 `config/links.yaml` 文件后重新执行 `npm run build`。

## 贡献指南

1. 在 GitHub 上 fork 本仓库，并克隆到本地开发环境。创建新分支时请使用 `feature/` 或 `fix/` 前缀，分支名简要描述修改内容。

2. 若需新增或修改外链数据，编辑 `config/links.yaml` 文件，遵循已有条目的格式（包含 `url`、`title`、`category`、`tags`、`description` 字段）。新增分类需同步更新 `config/site.yaml` 中的分类列表。

3. 代码修改完成后，在项目根目录执行 `npm run test` 确保所有单元测试与集成测试通过。新增功能请补充对应测试用例，测试文件存放于 `tests/` 目录下对应子目录。

4. 提交代码前请运行 `npm run lint` 进行代码风格检查（基于 ESLint 与 Prettier 配置），并确保 `npm run build` 可成功生成 `dist/` 目录。

5. 发起 Pull Request 到主仓库的 `main` 分支，描述中请清晰说明修改目的、影响范围以及是否涉及链接数据变动。PR 合并前 CI 流水线将自动执行构建与链接检测，如有失败请及时修复。

## 常见问题

**Q：links.yaml 中的链接数量较多时，构建速度会明显下降吗？**

A：HrefHub 构建时会对每条链接执行异步 HTTP HEAD 请求以检测可达性，该步骤是构建耗时的主要来源。对于超过 500 条链接的配置，建议在 `config/site.yaml` 中将 `validation.concurrency` 调整为 10 以下，或将 `validation.enabled` 设为 `false` 以跳过检测，仅在必要时手动触发检测脚本。

**Q：部署到静态托管平台后，如何更新链接数据？**

A：HrefHub 为纯静态生成器，所有数据在构建时固化。更新流程为：在本地修改 `config/links.yaml`，重新执行 `npm run build`，然后将生成的 `dist/` 目录重新部署至托管平台。推荐结合 CI/CD 服务（如 GitHub Actions）实现推送配置文件后自动构建与部署。

**Q：是否支持多语言界面？**

A：当前版本仅提供中文界面。开发者可在 `config/site.yaml` 中通过 `languages` 字段配置多语言文案映射，并在模板中使用 `{{ t 'key' }}` 辅助函数进行替换。完整的多语言支持计划在 v2.0.0 版本中正式提供。
