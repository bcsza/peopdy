# OpenLink Navigator

OpenLink Navigator 是一个专注于互联网优质资源聚合与导航的开源项目，旨在为技术研究者、内容创作者及普通网民提供一套高组织化、高可读性的外部资源索引体系。项目本身不托管任何实际内容，仅作为结构化链接仓库存在，通过对散布于网络各处的实用站点进行主题分类与注释说明，帮助用户快速定位所需工具、素材或信息入口。

本项目面向两类核心用户：一类是希望减少重复搜索成本、希望拥有个人定制化导航页面的终端用户；另一类是希望借鉴链接组织方式以搭建自身资源导航站点的开发者。通过简洁的 Markdown 文档与静态页面生成机制，OpenLink Navigator 将零散的书签转化为可维护、可分享、可版本追踪的开放式知识库。

## 功能概览

- 主题化分类索引：所有收录链接按使用场景与内容属性划分至独立章节，每个类别下提供链接用途的一行简述，便于用户按需浏览。

- 原始地址直出机制：项目严格保留用户提交的每一份原始 URL 字符串，不添加协议前缀、不修改域名大小写、不附加尾部斜杠，确保跳转路径与提交时完全一致。

- 轻量化静态结构：项目本身仅由单一 Markdown 文档与若干辅助脚本构成，无需数据库或后端服务，克隆后即可在本地以纯文本方式阅读或通过任意 HTTP 服务器托管。

- 可编程数据格式：除人类可读的表格与列表外，项目同时提供 JSON 格式的链接元数据导出，便于开发者将索引内容整合至浏览器插件、命令行工具或自动化巡检流程。

- 版本化更新日志：每次链接新增、删除或描述修改均通过 Git 提交记录体现，用户可通过提交历史追溯任意资源的引入时间与变更原因。

- 自定义扩展钩子：项目根目录预留 `custom` 文件夹，允许用户在不改动核心索引文件的前提下，通过外部 JSON 或 YAML 文件追加个人链接集合，避免与上游更新产生冲突。

- 链接可用性检查脚本：附带基于 curl 的简易健康检查工具，可定期对收录域名进行可达性测试，输出失效报告供维护参考。

## 应用场景

个人知识库的对外公开索引层。技术博主或笔记作者可将 OpenLink Navigator 作为其数字花园的“道路指示牌”，在文章或视频描述中引用该项目链接，让受众快速获取所用工具、参考文档或素材来源的统一入口，避免在评论区反复贴出散落链接。

小型团队内部资源协同。设计团队、运维小组或研究课题组可通过私有化部署的 OpenLink Navigator 实例，集中管理团队常用的设计素材站、监控面板地址、文献数据库入口等，新成员加入时只需克隆仓库即可同步全部外部资源认知。

离线文档辅助阅读环境。在受限网络环境下，用户可将本项目克隆至本地，配合项目提供的 `index.html` 生成脚本，构建一份全静态的本地导航页面，即使无外部网络，仍可通过本地文档查看所有链接描述与分类结构。

自动化监控系统的通知目标池。运维人员可将本项目中的监控面板链接提取后，配置到巡检脚本的告警通知模板中，确保每次告警消息内附带的跳转地址与项目索引保持同步，减少手动维护连接字符串的麻烦。

## 快速开始

以下命令序列适用于 Linux、macOS 及 Windows WSL 环境，请确保系统已安装 Git 与 Node.js（建议 LTS 版本）或 Python 3.8+。

```bash
# 克隆项目仓库至本地
git clone https://github.com/openlink-navigator/openlink-navigator.git
cd openlink-navigator

# 安装依赖（使用 npm 安装构建工具链）
npm install

# 执行本地构建，生成静态导航页面与 JSON 索引
npm run build

# 启动本地开发服务器，默认监听 8080 端口
npm start
```

若无需静态页面生成，仅作为 Markdown 索引阅读，则跳过构建步骤，直接使用任意 Markdown 阅读器打开 `README.md` 即可。对于 Python 用户，项目同时提供 `make.py` 脚本，执行 `python make.py --serve` 可启动简易 HTTP 服务器。

```
openlink-navigator/

├── package.json                  # npm 项目配置文件，定义依赖与脚本命令

│   ├── style.css                 # 生成页面的基础样式表，适配移动端与桌面端
│   ├── script.js                 # 前端交互脚本，提供分类折叠、搜索过滤功能

│   ├── build.js                  # 主构建脚本，读取数据并渲染模板输出 HTML

├── custom/                       # 用户自定义扩展目录（默认空，供本地覆盖）
│   ├── extra-links.json          # 示例自定义链接文件，用户可参照格式追加
│   └── custom-style.css          # 示例自定义样式文件，可覆盖默认样式
├── tests/                        # 单元测试目录
│   ├── schema.test.js            # 校验数据是否符合 JSON Schema
│   └── link-format.test.js       # 验证链接字符串是否满足原样输出规则

    └── workflows/                # GitHub Actions 工作流
        └── ci.yml                # 持续集成配置，自动执行构建与链接检查
```

## 贡献指南

我们欢迎任何形式的贡献，包括但不限于新增链接、修正描述、改进构建脚本或完善文档。请遵循以下步骤提交您的贡献：

1. 克隆项目至本地，并从 `main` 分支创建新的特性分支，分支命名建议采用 `feature/简述改动` 或 `fix/简述修复` 格式，避免直接在主干分支上操作。

2. 在 `data/links/` 对应分类的 JSON 文件中新增或修改链接条目。每个链接对象必须包含 `url`（原始字符串）、`title`（简短标题）和 `description`（一句话用途说明）三个字段，并确保 `url` 字段值与用户原始提交完全一致，不添加任何额外字符。

3. 提交前执行 `npm run test` 运行本地测试套件，确保所有数据校验通过且无格式错误。若新增分类，需同步更新 `data/categories.yaml` 文件并在 `data/links/` 下创建对应的 JSON 文件。

4. 推送分支至远程仓库，并通过 GitHub 界面发起 Pull Request。请在 PR 描述中清晰说明本次改动的动机、涉及链接列表以及是否已执行本地测试。维护者将在 48 小时内进行审核。

5. 若贡献涉及删除或替换已有链接，需在 PR 中额外提供失效证据（如 HTTP 状态码截图）或替代链接的合理性说明，否则维护者可能将请求标记为暂缓处理。

## 常见问题

Q: 为什么项目不直接托管图片、视频或文档文件，而只收集外部链接？

A: 本项目的核心定位是导航与索引，而非存储或分发具体内容。这样做有三点考量：一是规避版权与存储带宽风险，二是保持仓库轻量化和快速克隆体验，三是让用户始终访问来源最新版本，避免项目内缓存陈旧资源。所有链接均指向第三方站点，项目本身不承担内容可用性保证责任。

Q: 我发现某个收录链接已经失效或域名被劫持，应该如何处理？

A: 请通过 GitHub Issues 提交失效报告，标题注明 `[链接失效]` 并附上该链接的完整原始字符串以及您访问时收到的 HTTP 状态码或错误页面截图。维护者会定期使用 `scripts/check-links.sh` 进行全量检查，对于确认失效的链接，将在下一个版本中标记为 `[已失效]` 或直接移除。在提交报告前，建议您先使用在线 HTTP 状态检测工具进行双重确认。

Q: 我能否将本项目用于商业用途，例如嵌入公司内部导航系统或作为产品的一部分分发？

A: 可以。本项目采用 MIT 许可证，对使用场景不设限制，包括商业集成、再分发和私有修改。您唯一需要做的是保留项目原始版权声明。但需注意，本项目的 MIT 授权仅覆盖项目本身代码与文档结构，不延伸至所收录的外部链接指向的第三方内容，请自行遵守各目标站点的使用条款。
